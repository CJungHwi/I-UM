## 프로젝트 개요

이 프로젝트는 Next.js 16 기반의 풀스택 웹 애플리케이션으로, 다음과 같은 특징을 가지고 있습니다:

- **인증**: NextAuth.js를 사용한 JWT 기반 인증 시스템
- **데이터베이스**: MySQL Stored Procedure 전용 (mysql2/promise 직접 사용)
- **UI 프레임워크**: React 19 + Next.js 16 (App Router)
- **스타일링**: Tailwind CSS 4 + Shadcn UI
- **타입 안정성**: TypeScript

> **프레임워크 버전 (2026-04 기준)**: `package.json` 기준으로 Next.js **16.2.3**, React **19.2.5**, Tailwind **4.2.2** 등으로 정리되어 있습니다. 상세 버전은 아래 [기술 스택](#기술-스택)을 참고하세요.

## API 구조

### 1. API Routes (`src/app/api/`)

Next.js App Router의 API Routes를 사용합니다.

#### 인증 API (`src/app/api/auth/[...nextauth]/route.ts`)

```typescript
// NextAuth 핸들러를 export
export const { GET, POST } = handlers
```

- **경로**: `/api/auth/*`
- **역할**: NextAuth.js의 인증 엔드포인트 처리
- **메서드**: GET, POST
- **설명**: NextAuth.js가 자동으로 생성하는 인증 관련 엔드포인트들 (`/api/auth/signin`, `/api/auth/signout`, `/api/auth/session` 등)

---

### 2. Server Actions (`src/actions/`)

Next.js Server Actions를 사용해 서버 측 로직을 둘 수 있습니다. 공통 결과 타입은 `src/types/index.ts`의 **`ServerActionResult`** 를 참고합니다.

**현재 상태**: 비즈니스 전용 액션 파일은 제거된 상태이며, 새 기능 추가 시 `"use server"` 모듈을 이 디렉터리에 두면 됩니다.

---

### 3. 인증 시스템 (`src/auth.ts`, `src/auth.config.ts`)

#### `src/auth.config.ts`

NextAuth 설정 파일입니다.

**주요 설정**:
- **로그인 페이지**: `/login`
- **인가 콜백**: 대시보드 접근 시 인증 확인, 로그인된 사용자는 로그인 페이지 접근 시 홈으로 리다이렉트

#### `src/auth.ts`

NextAuth 인스턴스를 생성하고 설정합니다.

**주요 기능**:
- **세션 전략**: JWT (PrismaAdapter 미사용)
- **인증 제공자**: Credentials (ID/Password)
- **JWT 콜백**: 사용자 정보를 토큰에 저장
- **세션 콜백**: 토큰에서 사용자 정보를 세션에 매핑

**사용자 인증 프로세스**:
1. `mbId`와 `password`로 사용자 조회
2. `delYn: "N"` (삭제되지 않은 사용자만)
3. 비밀번호 검증 (PBKDF2-SHA256 또는 bcrypt)
4. JWT 토큰 생성 및 세션 저장

**Export**:
- `auth`: 서버 컴포넌트에서 세션 가져오기
- `signIn`: 로그인 함수
- `signOut`: 로그아웃 함수
- `handlers`: API 라우트 핸들러

---

### 4. 미들웨어 (`src/middleware.ts`)

**역할**: 모든 요청에 대해 인증 미들웨어 적용

**설정**:
- **매칭 경로**: `/((?!api|_next/static|_next/image|favicon.ico).*)`
- **제외 경로**: API, Next.js 정적 파일, favicon

**동작**:
- 인증되지 않은 사용자는 로그인 페이지로 리다이렉트
- 인증된 사용자는 요청된 페이지로 진행

**참고 (Next.js 16.2+)**:
- 빌드 시 `middleware` 파일 관련 **deprecation 경고**가 나올 수 있습니다. 공식 가이드에 따라 향후 **`proxy`** 규칙으로 이전하는 방안을 검토할 수 있습니다.
- 소스 파일은 **UTF-8**로 저장해야 합니다. 잘못된 인코딩의 주석 등은 Turbopack 파싱 오류를 유발할 수 있습니다.

### 5. 데이터베이스 라이브러리 (`src/lib/db.ts`)

**MySQL Connection Pool (mysql2/promise)**

**주요 변경 사항**:
- **Prisma ORM 제거**: 모든 ORM 기능 제거, 프로시저 호출 전용으로 전환
- **mysql2/promise 사용**: MySQL 드라이버를 직접 사용하여 프로시저 호출
- **Connection Pool**: 연결 풀을 사용하여 효율적인 연결 관리
- **서버 전용 모듈**: `server-only` 패키지로 클라이언트 번들링 방지

**주요 기능**:
- 동적 import로 `mysql2/promise`를 서버에서만 로드 (클라이언트 번들링 방지)
- 환경 변수로부터 DATABASE_URL 또는 분리된 변수로 연결 설정 생성
- Connection Pool 설정: 최대 10개 연결, Keep-Alive 활성화

**환경 변수 지원**:
- `DATABASE_URL`: 직접 설정 가능
- 또는 분리된 변수: `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`

**주요 함수**:

1. **`callProcedure<T>(procedureName: string, ...params: any[]): Promise<T[]>`**
   - MySQL Stored Procedure 호출
   - SQL 인젝션 방지를 위한 파라미터 바인딩
   - 프로시저 결과의 첫 번째 결과셋 반환

2. **`executeQuery<T>(query: string, params?: any[]): Promise<T[]>`**
   - 일반 SQL 쿼리 실행 (필요한 경우)
   - 파라미터 바인딩 지원

3. **`closePool(): Promise<void>`**
   - 연결 풀 종료 (앱 종료 시 사용)

**사용 예시**:
```typescript
import { callProcedure } from "@/lib/db"

// 프로시저 호출
const results = await callProcedure<any>(
    "get_menu",
    userLevel
)

// 결과 매핑 (프로시저 결과는 컬럼명 또는 f0, f1, f2... 형태로 반환될 수 있음)
const mappedResults = results.map((item: any) => ({
    id: item.id ?? item.f0 ?? 0,
    name: item.name ?? item.f1 ?? "",
    // ...
}))
```

**주의사항**:
- `db.ts`는 `server-only`로 마킹되어 있어 클라이언트 컴포넌트에서 직접 import 불가
- 모든 DB 호출은 프로시저를 통해 수행
- 프로시저 결과는 컬럼명 또는 `f0`, `f1`, `f2`... 형태로 반환될 수 있으므로 둘 다 처리해야 함

### 6. 메뉴 라이브러리 (`src/lib/menu.ts`, `src/lib/menu-utils.ts`)

**파일 분리 구조**:
- **`menu.ts`**: 서버 전용 (DB 호출 함수)
- **`menu-utils.ts`**: 클라이언트/서버 공통 (순수 유틸리티 함수)

**주요 변경 사항**:
- 클라이언트 컴포넌트에서 `menu.ts`를 직접 import하면 `mysql2`가 번들링되는 문제 해결
- 순수 함수(`buildMenuHierarchy`, `getIconComponent`)를 `menu-utils.ts`로 분리
- `menu.ts`에 `server-only` 마커 추가

**`src/lib/menu.ts` (서버 전용)**:

1. **`getMenuItems(userLevel?: number | null)`**
   - 사용자 레벨에 따라 메뉴 조회
   - MySQL 프로시저 `get_menu()` 호출
   - React.cache()로 중복 쿼리 방지
   - 프로시저 결과를 `MenuItem` 타입으로 매핑

**`src/lib/menu-utils.ts` (클라이언트/서버 공통)**:

1. **`buildMenuHierarchy(items: MenuItem[]): NavItem[]`**
   - 평면 메뉴 배열을 계층 구조로 변환
   - `sortOrder` 기준으로 정렬
   - 클라이언트 컴포넌트에서도 사용 가능

2. **`getIconComponent(iconName: string | null)`**
   - 아이콘 이름 문자열을 Lucide React 컴포넌트로 변환
   - 클라이언트 컴포넌트에서도 사용 가능

**사용 예시**:
```typescript
// 서버 컴포넌트
import { getMenuItems, buildMenuHierarchy } from "@/lib/menu"

const menuItems = await getMenuItems(userLevel)
const navItems = buildMenuHierarchy(menuItems)

// 클라이언트 컴포넌트
import { getIconComponent } from "@/lib/menu-utils"

const Icon = getIconComponent("Home")
```

### 4. 전역 스타일 (`src/app/globals.css`)

| 반경 | `--radius-sm` / `--radius-md` / `--radius-lg` (`--radius` 기준 계산) |

### 데이터베이스 설정

**프로시저 전용 아키텍처**:
- Prisma ORM을 제거하고 MySQL Stored Procedure만 사용
- 모든 데이터베이스 작업은 프로시저를 통해 수행

- 프로시저 파일 위치: `prisma/procedures/`
- 프로시저는 데이터베이스에 직접 생성되어 있어야 함
- 프로시저 생성/수정은 데이터베이스 관리 도구를 통해 수행

**앱 코드에서 직접 호출하는 프로시저 (현재)**:
- `get_menu(userLevel)`: 사이드바 메뉴 (`src/lib/menu.ts`)
- `get_user_by_id(mb_id)`: 로그인 사용자 조회 (`src/auth.ts`)

**DB에 존재할 수 있는 기타 프로시저** (과거 기능·배치·직접 호출 등):
- 예: 가격/URL/SKU 관련 프로시저는 DB·운영 정책에 따라 유지될 수 있으나, 현재 Next 앱 라우트에서는 호출하지 않을 수 있음. 신규 기능 설계 시 `callProcedure` 패턴만 동일하게 적용하면 됨.

**프로시저 호출 패턴**:
```typescript
import { callProcedure } from "@/lib/db"

// 프로시저 호출
const results = await callProcedure<any>("procedure_name", param1, param2)

// 결과 매핑 (컬럼명 또는 f0, f1, f2... 형태 지원)
const mapped = results.map((item: any) => ({
    id: item.id ?? item.f0 ?? null,
    name: item.name ?? item.f1 ?? "",
    // ...
}))
```

## 데이터베이스

### 아키텍처 변경 사항

**Prisma ORM 제거 → Stored Procedure 전용**

이 프로젝트는 Prisma ORM을 완전히 제거하고 MySQL Stored Procedure만 사용하는 아키텍처로 전환되었습니다.

**변경 이유**:
- 모든 데이터베이스 작업을 프로시저로 통일하여 보안 강화
- SQL 인젝션 방지 및 쿼리 최적화를 데이터베이스 레벨에서 관리
- ORM 오버헤드 제거 및 성능 향상

**주요 변경 사항**:
1. `prisma/schema.prisma` 파일 제거
2. `@prisma/client`, `prisma` 패키지 제거
3. `mysql2/promise` 직접 사용
4. 모든 DB 호출을 `callProcedure()` 함수로 통일
5. `server-only` 패키지로 서버 전용 모듈 마킹

### 데이터베이스 테이블 구조

**주요 테이블**:

1. **`g5_member`**
   - 사용자 정보
   - 인증에 사용되는 필드: `mb_id`, `mb_password`, `mb_level`, `del_yn`

2. **`auto_sku_target_url`**
   - 타겟 URL 관리

3. **`auto_sku_search_url`**
   - 검색 URL 관리
   - 상태: `pending`, `A` (승인), `S` (중지), `R` (대기)

4. **`auto_sku_daily_price`**
   - 일일 가격 데이터

5. **`auto_sku`**
   - SKU 설정
   - 필드: `wr_id`, `sku_exec_yn`, `Discount_rate`, `min_price`

6. **`auto_config`**
   - 시스템 설정

7. **`ium_menu`**
   - 메뉴 계층 구조
   - 권한 레벨 기반 필터링 지원

8. **`g5_write_product`**
   - 상품 정보
   - 필드: `wr_id`, `wr_num`, `wr_subject`, `wr_1` (SKU), `wr_name`, `wr_3`, `wr_33`, `wr_delYn`

### Stored Procedures

**프로시저 파일 위치**: `prisma/procedures/`

**주요 프로시저**:

1. **`get_menu(userLevel)`**
   - 사용자 레벨에 따라 메뉴 조회
   - 계층 구조 및 권한 필터링
   - 반환: 메뉴 목록 (컬럼명 또는 f0, f1, f2... 형태)

2. **`get_user_by_id(mb_id)`**
   - 사용자 정보 조회 (인증용)
   - 반환: 사용자 정보

3. **기타 비즈니스 프로시저** (예시)
   - 과거 앱 기능에서 사용하던 SKU·URL·일일가격 등 관련 프로시저는 DB에 정의되어 있을 수 있음
   - 현재 저장소의 Next 라우트에서 직접 호출하지 않을 수 있음 → 신규 화면에서 필요 시 `callProcedure`로 연동

**프로시저 호출 예시**:
```typescript
import { callProcedure } from "@/lib/db"

// 프로시저 호출
const results = await callProcedure<any>(
    "get_menu",
    userLevel
)

// 결과 매핑 (컬럼명 또는 f0, f1, f2... 형태 모두 지원)
const mappedResults = results.map((item: any) => ({
    id: item.id ?? item.f0 ?? 0,
    menuId: item.menu_id ?? item.f1 ?? "",
    title: item.title ?? item.f2 ?? "",
    // ...
}))
```

**프로시저 결과 처리**:
- 프로시저 결과는 컬럼명 또는 `f0`, `f1`, `f2`... 형태로 반환될 수 있음
- 안전한 매핑을 위해 `??` 연산자를 사용하여 둘 다 처리
- 예: `item.id ?? item.f0 ?? null`

## 마이그레이션 가이드 (Prisma → Stored Procedure)

### 변경된 파일 목록

**제거된 파일**:
- `prisma/schema.prisma` (Prisma 스키마 파일)

**변경된 파일**:
- `src/lib/db.ts`: PrismaClient → mysql2/promise Connection Pool
- `src/lib/menu.ts`: 서버 전용으로 변경, 순수 함수 분리
- `src/lib/menu-utils.ts`: 새로 생성 (순수 유틸리티 함수)
- `src/auth.ts`: Prisma → 프로시저 호출로 변경
- (과거) API/페이지/Server Actions는 Prisma → 프로시저 패턴으로 정리됨. 현재는 비즈니스 API·페이지가 비어 있을 수 있음.

**추가된 파일**:
- `prisma/procedures/*.sql`: 프로시저 정의 파일들
- `src/lib/menu-utils.ts`: 클라이언트/서버 공통 유틸리티

### 코드 변경 패턴

**이전 (Prisma)**:
```typescript
import { prisma } from "@/lib/db"

const users = await prisma.user.findMany({
    where: { delYn: "N" }
})
```

**변경 후 (Stored Procedure)**:
```typescript
import { callProcedure } from "@/lib/db"

const results = await callProcedure<any>("get_users")
const users = results.map((item: any) => ({
    id: item.id ?? item.f0 ?? 0,
    name: item.name ?? item.f1 ?? "",
    // ...
}))
```

### 주의사항

1. **서버 전용 모듈**: `db.ts`와 `menu.ts`는 `server-only`로 마킹되어 클라이언트에서 직접 import 불가
2. **결과 매핑**: 프로시저 결과는 컬럼명 또는 `f0`, `f1`, `f2`... 형태로 반환될 수 있으므로 둘 다 처리해야 함
3. **타입 안정성**: 프로시저 결과는 `any` 타입으로 반환되므로 매핑 시 타입을 명시적으로 지정해야 함
4. **에러 처리**: 프로시저 호출 실패 시 적절한 에러 처리 필요
